Scan vulnerabilites of non-OS packages

オプション

non-OS-packagesの脆弱性をスキャンするために、以下のオプションが用意されています。

  • lockファイルの指定(ライブラリ)
  • GitHubとの連携(ライブラリ)
  • config.tomlへのCPEの設定(ネットワークデバイス、ライブラリ)
  • OWASPの依存性チェック(ライブラリ)

ライブラリの脆弱性のスキャン

Vulsは、v0.8.0からknqyf263/trivyを用いてライブラリをスキャン出来ます。Trivyは、以下のlockファイルを検知します。

  • Gemfile.lock
  • Pipfile.lock
  • poetry.lock
  • composer.lock
  • package-lock.json
  • yarn.lock
  • Cargo.lock

以下はconfig.tomlの例です。
lockファイルのパスを指定することで、CPEを指定せずにライブラリの脆弱性を検知します。

[servers]

[servers.abuntu]
host         = "xxx.xxx.xxx"
port        = "22"
user        = "tamachi"
keyPath     = "/Users/amachi/.ssh/id_dsa"
findLock = true # lockファイルの自動検知
lockfiles = [
  "/home/tamachi/lockfiles/package-lock.json",
  "/home/tamachi/lockfiles/yarn.lock"
]

使い方: GitHub Security Alertsとの連携

GitHubは、リポジトリの依存先の報告された脆弱性を追跡し、影響があるリポジトリにセキュリティアラートを提供します。 GitHub Security Alerts. GitHubにより検知された脆弱性はGitHubのAPIを使用してインポートできます。

[servers.serverA]
user = "root"
host = "10.0.0.1"
port = "22"
keyPath = "/Users/kanbe/.ssh/id_rsa"
scanMode = ["fast-root"]

[servers.serverA.githubs."owner/repo"]
token   = "xxxxYourTokenxxx"

GitHub上で無視した脆弱性を無視するために、 -ignore-github-dismissedオプションが用意されています。

$ ./vuls report -h
report:
        report
               ...
                [-ignore-github-dismissed]
               ...

  -ignore-github-dismissed
        Don't report the dismissed CVEs on GitHub Security Alerts
  ...

CPEのスキャン

自分でコンパイルしたパッケージや、言語のライブラリ、フレームワークなど、CPEに登録されたnon-OS パッケージの脆弱性を検知することもできます。

ソフトウェアの名前からCPEを検索する方法

NVD: Search Common Platform Enumerations (CPE) Check CPE Naming Format: 2.2が必要です。

詳しい人はgo-cpe-dictionaryを使用すると良いでしょう。

CPEをアプリケーションの名前でインクリメンタルに検索できます。

設定

ホストOS

Ruby on Rails v4.2.1とPostgreSQL9.6.2の脆弱性を検知するためには、cpeNamesをサーバのセクションに設定する必要があります。

[servers]

[servers.172-31-4-82]
host         = "172.31.4.82"
user        = "ec2-user"
keyPath     = "/home/username/.ssh/id_rsa"
cpeNames = [
    "cpe:/a:rubyonrails:ruby_on_rails:4.2.1",
    "cpe:2.3:a:postgresql:postgresql:9.6.2:*:*:*:*:*:*:*",
]

コンテナ

特定のコンテナの中のRuby on Rails v4.2.1の脆弱性を検知するためには、サーバの中のコンテナのセクションにcpeNamesを設定する必要があります。 以下はdockerAの中のRuby on Rails v4.2.1とPostgreSQL9.6.2の脆弱性を検知する例です。

[servers]

[servers.172-31-4-82]
host         = "172.31.4.82"
user        = "ec2-user"
keyPath     = "/home/username/.ssh/id_rsa"
containerType = "docker"
containersIncluded = ["${running}"]

[servers.172-31-4-82.containers.dockerA]
cpeNames = [
    "cpe:/a:rubyonrails:ruby_on_rails:4.2.1",
    "cpe:2.3:a:postgresql:postgresql:9.6.2:*:*:*:*:*:*:*",
]

type="pseudo"

CPEを指定してSSH接続無しで脆弱性を検知するためには、これを指定します。 スキャンの時には、pseudo typeは何もしません。 レポートのタイミングでNVDを検索し、指定されたCPEのソフトウェアの脆弱性を検知します。

[servers]

[servers.forti]
type = "pseudo"
cpeNames = [
    "cpe:/o:fortinet:fortios:4.3.0",
]

使い方: ライブラリがアップデートされたときに自動でアップデートするために、OWASP Dependency Checkと連携（実験中）

OWASP Dependency checkは、そのプロジェクトの依存を特定して、既知の公開済みの脆弱性があるかをチェックするツールです。

VulsとOWASP Dependency Checkを連携する利点は以下の通りです。

  • ライブラリがアップデートされたときに、Vulsのコンフィグを自動で更新する。
  • Vulsの機能を用いてEmailやSlackで通知する。
  • 日本語でのレポートができる。
    • OWASP Dependency Checkは英語しか対応していません。

VulsとOWASP Dependency Checkを連携させる方法は以下の通りです。

  • OWASP Dependency Checkを、--format=XMLをつけて実行する

  • そのXMLをconfig.toml内で以下のように定義する

    ```bash
[servers]

[servers.172-31-4-82]
host         = "172.31.4.82"
user        = "ec2-user"
keyPath     = "/home/username/.ssh/id_rsa"
owaspDCXMLPath = "/tmp/dependency-check-report.xml"
```

以下に、あるコンテナに対するOWASP Dependency CheckのXMLを指定する方法を示します。

[servers]

[servers.172-31-4-82]
host         = "172.31.4.82"
user        = "ec2-user"
keyPath     = "/home/username/.ssh/id_rsa"
containerType = "docker"
containersIncluded = ["${running}"]

[servers.172-31-4-82.containers.dockerA]
owaspDCXMLPath = "/tmp/dependency-check-report.xml"
